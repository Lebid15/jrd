# خطة تطوير مشروع JRD — التحوّل إلى نظام جرد آلي شامل

> الهدف: تحويل نظام الجرد اليومي الحالي من إدخال يدوي إلى نظام **آلي بالكامل** عبر مزيج من: روبوتات تصفّح (Web Scraping/Automation)، تكاملات API، وبوت واتساب (Baileys). ثم إضافة طبقة **جرد شهري** فوق الجرد اليومي.

> **منهجية النقاش**: نناقش كل بند مستقلاً ونتفق على التفاصيل والـ schema قبل كتابة أي كود. كل بند سيُضاف إليه قسم تفصيلي لاحقاً في هذا الملف.

> ⚠️ **ملاحظة دائمة**: هذا الملف يُحدَّث باستمرار مع كل خطوة ننجزها. كل إنجاز يُعلَّم ✅ وكل تفصيل جديد يُضاف في مكانه.

## 🟡 البند 4 — بوت واتساب (Baileys) — منجز ✅

### ما أُنجز ✅
- مجلد `bot/` كامل (Node + Baileys + Express على port 3100).
- `session.js` — كلاس يدير اتصال واتساب (QR → connected → reconnect تلقائي).
- `sessionManager.js` — يستأنف الجلسات تلقائياً عند إعادة التشغيل.
- `encryptedAuthStore.js` — تشفير AES-256-GCM لملفات جلسة Baileys.
- `server.js` — HTTP داخلي: start / logout / status / sessions.
- `backendClient.js` — يرسل الرسائل الواردة إلى `/api/internal/ingest`.
- `backend/src/routes/internal.js` — routes داخلية للواجهة والبوت.
- جدول `whatsapp_messages` + `whatsapp_transactions` في DB.
- صفحة **واتساب** في الواجهة: حالة + QR + إدارة المجموعات + الكلمات المفتاحية + أداة اختبار + آخر الرسائل.
- `start.sh` يشغّل backend والبوت معاً.
- متغيّرات Railway: `BOT_URL` + `INTERNAL_API_KEY` + `AUTH_DIR=/data/auth_sessions`.
- إصلاح تشخيص الحالة: التفريق بين `idle`/`auth_mismatch`/`offline`.

## ✅ البند 8 — الجرد الشهري — منجز

### الفكرة
- زرّ جديد في **الجرد اليومي** بجوار "حفظ الجرد" اسمه **"جرد شهري"** (لون بنفسجي).
- قسم جديد في السايدبار **"الجرد الشهري"** (`/monthly`).

### الآلية
- **أوّل مرّة**: snapshot لرأس المال الحالي فقط، `period_profit = 0`.
- **المرّات التالية**: snapshot جديد + جمع `profit` لكل جرد يومي تمّ بعد `created_at` لآخر جرد شهري وحتّى الآن.
- يحفظ snapshot كامل للبنود (مثل الجرد اليومي) → قابل للعرض/التصدير لاحقاً.

### Schema الجديد
- جدول `monthly_inventories`:
  `id, date, exchange_rate, total_try, total_usd, total_converted_usd,
   previous_monthly_id, previous_total_usd, period_from, period_to,
   period_profit, daily_count, notes, created_at`
- جدول `monthly_inventory_items`: نفس بنية `inventory_items`.

### API
- `GET /api/monthly` — قائمة + total
- `GET /api/monthly/:id` — تفاصيل + بنود
- `GET /api/monthly/last/one` — الأخير
- `GET /api/monthly/preview/next` — معاينة قبل الحفظ (is_first, period_profit, daily_count, ...)
- `POST /api/monthly` — حفظ جرد شهري جديد
- `DELETE /api/monthly/:id`

### الواجهة
- صفحة `MonthlyArchive.jsx`: بطاقة معاينة + قائمة جرود قابلة للطيّ (تفاصيل + جدول البنود) + تصدير PDF + حذف.
- زرّ في `Dashboard.jsx` يعرض تأكيداً واضحاً قبل الحفظ.

## البنود الأساسية (Roadmap)

### 1. روبوت موقع شحن الألعاب (الخاص بنا)
- **الوصف**: موقعنا الذي يبيع شحنات الألعاب يحتوي صفحة تعرض جميع المحلات الموزّعة لدينا مع أرصدة كل محل + المجموع النهائي.
- **المطلوب**: روبوت يسجّل دخول إلى الموقع، يفتح صفحة المحلات، يجلب: `[اسم المحل، الرصيد، (مديونية إن وُجدت)]` + المجموع الكلي.
- **الإخراج**: تحديث بنود `type='shop'` (نوع جديد) في الجرد اليومي تلقائياً، وحفظ snapshot في جدول مستقل للأرشيف.
- **التقنية المرشّحة**: Playwright (أقوى من Puppeteer للمواقع المحميّة)، خدمة Node منفصلة `scraper/` أو داخل خدمة البوت نفسها.
- **نقاش لاحق**: عنوان الموقع، طريقة الدخول (user/pass/2FA؟)، شكل الجدول في الصفحة، تكرار الجلب (كل ساعة؟ عند الطلب؟).

---

### 2. روبوت موقع شحن المتابعين/المشاهدات (طرف ثالث)
- **الوصف**: موقع منفصل لا نملكه، نشتري منه خدمات SMM، نحتاج فقط **رصيدنا لديهم**.
- **المطلوب**: روبوت يسجّل دخول دورياً، يقرأ قيمة الرصيد فقط، يحدّث بنداً واحداً في الجرد.
- **الإخراج**: تحديث `current_values` لبند مرتبط `type='smm_provider'` (أو إعادة استخدام `provider` مع نوع جديد).
- **نقاش لاحق**: هل لديهم API رسمي؟ (أغلب لوحات SMM لديها API بمفتاح — يُفضّل API على scraping). اسم الموقع، آلية الدخول.

---

### 3. روبوت البنوك (Bank Scraping)
- **الوصف**: جلب أرصدة الحسابات البنكية تلقائياً.
- **المطلوب**: روبوت/روبوتات لكل بنك (قد يحتاج كل بنك driver منفصل) → قراءة الرصيد الحالي → تحديث بند `type='bank'`.
- **التحديات الأمنية**:
  - 2FA/OTP غالباً مطلوب → قد يحتاج تدخل يدوي أو ربط بـ SMS.
  - خطر إقفال الحساب لو اكتُشف automation.
  - تشفير كلمات السر في القاعدة (مثل ما فعلنا مع `auth_sessions`).
- **نقاش لاحق**: ما البنوك بالاسم؟ هل تدعم Open Banking API؟ هل يكفي تحديث يومي واحد قبل الجرد؟

---

### 4. بوت واتساب — البنية الأساسية (Baileys)
- **الوصف**: خدمة Node منفصلة بنفس المعمارية الموثّقة في [bothelp.md](bothelp.md).
- **المطلوب**:
  - إنشاء مجلد `bot/` (Node + Express + Baileys + AES-256-GCM لجلسات).
  - مفتاح مشترك `INTERNAL_API_KEY` بين البوت و backend.
  - single-tenant مبدئياً (`tenant_id="1"` ثابت) مع الإبقاء على بنية المسارات multi-tenant.
  - نقاط backend داخلية: `/api/internal/ingest`, `/api/internal/command`, `/api/internal/reconcile-log`.
  - صفحة واجهة: ربط واتساب (QR) + حالة الاتصال + logout.
- **نقاش لاحق**: رقم البوت الثانوي، اسم البوت الظاهر في "الأجهزة المرتبطة".

---

### 5. مطابقة الصرافين (Reconciliation — Exchangers)
- **الوصف**: لدينا صرافون نحن (نحوّل لهم) وصرافون لكم (يحوّلون لنا). كل عملية يجب أن **تتطابق** بين طرفين (نحن + الصراف) عبر رسائل واتساب.
- **المطلوب**:
  - تعريف "صراف" ككيان في DB مع تفاصيل (الاسم، المجموعة، العملة، النمط).
  - بوت يستقبل رسائل من مجموعة كل صراف ويستخرج: `[المبلغ، الاتجاه (نا/منا)، الوقت، المرجع]`.
  - شاشة مطابقة: قائمة بالعمليات غير المطابَقة، مع زر mark-as-matched يدوي + matching تلقائي بالمبلغ والوقت.
  - تنبيه بالفروقات (شبهة خطأ/تأخير/سرقة).
- **نقاش لاحق**: عدد الصرافين، شكل رسائلهم النموذجي، فترة السماح للمطابقة (±5 دقائق؟).

---

### 6. مطابقة العامل الميداني — مجموعة "غلة"
- **الوصف**: عامل يجمع نقداً من المحلات يدوياً ويكتب في مجموعة واتساب (اسمها "غلة") مثل: `استلمت 1500 ليرة من محل X`. لاحقاً يُسلّم النقود لصراف يكتب: `استلمت من [اسم العامل] 1500`.
- **المطلوب**:
  - parser لمجموعة "غلة" → سجل "تحصيلات العامل" `[المحل، المبلغ، التاريخ، حالة التسليم]`.
  - parser لمجموعات الصرافين → سجل "تسليمات للصراف".
  - مطابقة: مجموع تحصيلات العامل في الفترة = مجموع تسليماته للصرافين.
  - شاشة "كشف العامل" + تنبيه عند فرق.
- **نقاش لاحق**: أسماء العمال، أسماء المحلات، صيغة الرسائل النموذجية، هل العامل واحد أم عدة عمّال؟

---

### 7. نوعا الحوالات من المحلات
- **الوصف**: المحلات تُسدّد لنا بطريقتين:
  - **(أ) تحويل بنكي/أونلاين** → يظهر في بند البنوك (يُعالج في البند 3).
  - **(ب) نقد عبر العامل** → يظهر في كشف العامل (يُعالج في البند 6).
- **المطلوب**: ربط منطقي بين الطريقتين على مستوى **المحل**:
  - جدول `shops` يحتوي كل المحلات.
  - كل تحويل/تحصيل يُربط بـ `shop_id`.
  - شاشة "كشف محل" تجمع: ما عليه من شحن (من البند 1) + ما دفعه (بنكي + نقدي) = الرصيد المستحق.
- **نقاش لاحق**: هل المحلات في موقع شحن الألعاب (البند 1) هي نفسها التي تدفع نقداً، أم منفصلة؟ كيف نوحّد الأسماء؟

### 8. الجرد الشهري
- **الوصف**: قسم جديد في السايدبار بجانب "الجرد" الحالي (اليومي).
- **المطلوب**:
  - حساب آلي شهري بناءً على كل جرد يومي محفوظ خلال الشهر.
  - مؤشرات: إجمالي الربح الشهري، أعلى/أقل يوم، متوسط الربح اليومي، عدد أيام الخسارة.
  - تفصيل مصادر الإيرادات (شحن ألعاب / SMM / فروقات صرف).
  - تفصيل المصاريف (إن قرّرنا إضافة جدول `expenses`).
  - تقرير PDF شهري (مثل تقرير الأرشيف الحالي لكن مجمّع).
  - مقارنة بالشهر السابق.
- **نقاش لاحق**: ما الحقول الإضافية المطلوب رصدها لتمكين التقرير الشهري؟ هل نحتاج جدول `expenses`؟ هل نريد رسوماً بيانية (Recharts)؟

### 9. إعادة هيكلة قاعدة البيانات
- **الوصف**: مع كل التغييرات أعلاه، schema الحالي ضيّق.
- **الجداول الجديدة المتوقّعة** (مبدئياً، نُنقّحها في كل بند):
  - `shops` (المحلات: id, name, source_url_id, ...)
  - `workers` (العمّال الميدانيون)
  - `exchangers` (الصرافون)
  - `bank_accounts` (الحسابات البنكية)
  - `transactions` (كل عملية مالية: نوع، اتجاه، طرف، مبلغ، عملة، حالة مطابقة)
  - `whatsapp_groups` (تعريف المجموعات وربطها بـ entity)
  - `whatsapp_messages` (أرشيف خام للتشخيص)
  - `parse_rules` (قواعد regex لاستخراج البيانات من الرسائل)
  - `scrape_jobs` + `scrape_runs` (سجلّ تنفيذ الروبوتات)
  - `monthly_reports` (تقارير شهرية مُحفوظة)
- **نقاش لاحق**: نأخذ جدولاً واحداً مع كل بند بدل دفعة واحدة.
